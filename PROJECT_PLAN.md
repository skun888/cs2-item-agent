# CS2 Item Agent 开发思路与工程规范

> 状态：阶段 0 至阶段 8 已完成；进入真实使用反馈与发布准备
> 更新日期：2026-07-22
> 项目性质：免费开源、本地优先、模型无关

## 1. 产品定位

CS2 Item Agent 是一个免费开源、在用户本地运行的 CS2 饰品市场决策助手。

用户只需配置自己的 SteamDT API Key，即可通过 Codex、Qoder、Trae 等支持本地命令或 MCP 的 Agent 工具：

- 查询 SteamDT 聚合的多平台价格、在售量、求购价、求购量和 K 线；
- 监控指定公开 SteamID 的库存变化；
- 分析市场供需、板块强弱、库存异动和价格背离；
- 评估 CS2 饰品经过七日交易保护期后的挂刀/搬砖风险调整收益；
- 根据枪皮、预算和目标风格推荐贴纸搭配，并生成模拟预览或检视方案；
- 在用户主动启用监控时，通过企业微信通知库存和市场异动。

本项目不是网站、交易平台、自动交易机器人或全网持有人数据库。产品以按需查询和对话决策为主，持续 Worker 属于用户主动开启的可选能力。Agent 工具只是入口，核心能力统一由本地 CLI、MCP 服务和可选 Worker 提供。

## 2. 核心价值

现有免费工具普遍能够展示价格、库存估值和图表。本项目的差异不是重复制作行情页面，而是：

1. 将公开库存变化与多平台行情放在同一证据链中分析；
2. 用自然语言回答用户的具体决策问题；
3. 将七日不可转移期间的价格、流动性和成本风险纳入挂刀判断；
4. 把计算、证据、市场知识和语言解释分层，避免模型编造事实；
5. 利用 SteamDT 已有历史 K 线立即提供基础分析，并在用户本机积累补充快照，不建设中心用户数据库。

## 3. 已确认的产品原则

### 3.1 本地优先

- 默认使用 SQLite，建议路径为 `data/cs2-item-agent.db`；
- API Key 仅从环境变量或本地密钥配置读取；
- 用户库存、监控任务、历史快照和通知记录默认不上传；
- GitHub 仓库不包含真实 API Key、Cookie、Token 或未脱敏 SteamID；
- 提供 `.env.example`，不得提交真实 `.env`；
- 示例数据必须使用虚构账号或不可逆哈希标识。
- 第一版不包含任何匿名遥测，不向项目维护者发送版本、使用次数或错误统计。

### 3.2 模型无关

- 核心能力通过 CLI 和 MCP 工具暴露；
- 不绑定某个模型、Agent 客户端或云端推理服务；
- 核心程序负责获取、校验、计算和返回结构化结果；
- 知识库负责提供市场分析框架和表达规范；
- 模型只负责理解问题、选择工具、组织语言和辅助判断；
- 没有模型时，CLI 仍应能输出完整、可验证的结构化报告。

### 3.3 证据优先

所有分析输出至少包含：

- 数据来源；
- 数据时间；
- 数据新鲜度；
- 使用的关键指标；
- 证据等级；
- 置信度；
- 缺失数据与限制。

证据等级统一为：

| 等级 | 含义 | 示例 |
| --- | --- | --- |
| 已验证事实 | 数据源直接证明 | 某公开库存快照中出现一件饰品；平台接口返回当前最低价 |
| 高置信推断 | 多项连续证据支持，但缺少直接交易记录 | 某账号库存减少，同时市场在售量明显增加 |
| 未知 | 数据缺失、库存私密、请求失败或证据冲突 | 无法判断饰品是出售、转仓还是库存转私密 |

公开库存中饰品消失不能直接称为“卖出”；账号之间出现相同饰品只能称为“流转线索”；关系图只能表示饰品交易关联，不代表现实好友或社会关系。

## 4. 第一阶段核心功能

### 4.1 SteamDT 市场数据与行情分析

第一版默认使用 SteamDT，公开能力以实际文档和脱敏成功响应为准，包括：

- 饰品基础信息；
- 单个和批量多平台价格；
- 最低在售价与在售数量；
- 最高求购价与求购数量；
- 饰品 K 线；
- 所有平台近 7 天均价；
- 市场大盘 K 线；
- 市场大盘最新指数；
- 检视链接磨损解析；
- 检视图生成。

已知限制：

- SteamDT 当前饰品 K 线文档明确说明不包含成交数据；
- SteamDT 已有历史 K 线可以立即用于价格趋势、波动率和历史滚动七日收益，基础分析不需要等待本地数据冷启动；
- 本地快照主要补充历史在售量、求购量、跨平台价差、公开库存变化，以及 Agent 信号的七日后复盘；
- K 线请求中的 `type` 含义、最长历史范围、平台枚举和返回点结构未在公开文档中说明，需要真实 API 响应验证；
- 不得将 K 线推断出的活跃度描述为真实成交量；
- Steam 官方市场公开 `priceoverview` 当前可返回最低价、中位价和近期成交量，可作为低频、可缓存的辅助数据源；该接口未正式文档化，不能作为唯一可靠来源；
- 数据可用性、频率限制和免费政策可能变化；
- SteamDT 是第一版默认数据源，但核心必须保留可替换的适配器接口。

市场分析指标第一版包括：

- 多平台价格差；
- 买卖价差；
- 在售量与求购量变化；
- 价格与供需背离；
- 7/14/30 日收益与波动；
- 单品相对大盘强弱；
- 当前价格历史分位；
- 历史滚动七日收益分布；
- 流动性评分；
- 数据新鲜度和完整度。

### 4.2 公开 Steam 库存监控

用户提交一个或多个 SteamID，系统可以按需获取公开库存并生成追加式快照。只有用户主动开启监控时，才按计划持续执行；默认监控间隔为 30 分钟且允许配置。

第一版支持：

- 新增和移除饰品；
- 同类饰品数量变化；
- 指定饰品或板块监控；
- 高价值饰品转入和转出；
- 库存多平台估值变化；
- 可获得时保存磨损、模板、贴纸、名称标签和 StatTrak/Souvenir 信息；
- 按首次观察、最后观察构建本地时间线；
- 企业微信 Webhook 告警。

系统必须区分以下状态：

- 库存公开且成功获取；
- 库存公开但请求失败或被限流；
- 库存私密或不可见；
- 返回数据不完整；
- 与上次快照相比发生变化。

请求失败或库存变为不可见时，不得生成“饰品已全部移除”的错误事件。

### 4.3 七日挂刀/搬砖风险评估

本功能定位为决策支持，不是自动交易或收益保证。

基础计算：

```text
七日后净收入
= 七日后目标平台预计可成交价
- 卖出手续费
- 提现或折现成本
- 预计滑点

七日后净利润
= 七日后净收入
- 当前实际买入成本
- 其他可配置成本
```

评估至少考虑：

- 当前跨平台价差；
- 买入和卖出手续费；
- 余额折现、提现成本；
- 在售量、求购量和买卖价差；
- 近期收益、波动率和最大回撤；
- 跨平台价差是否正在收敛；
- 单品相对大盘强弱；
- 被监控公开账号的相关库存变化；
- 数据缺失、陈旧和接口异常。

第一版可以直接利用 SteamDT 历史 K 线做初步挂刀分析：计算历史滚动七日收益、波动率、最大回撤和相对大盘强弱，再结合当前跨平台价差与用户费率模板生成三种情景。本地长期数据只用于增强供求变化判断和校准历史建议，不是初步分析的前置条件。

输出不只给一个预测价格，而是提供：

- 乐观、基准、悲观三种七日情景；
- 每种情景的扣费后收益；
- 盈利概率或历史可比成功率（只有经过回测后才可输出）；
- 流动性等级；
- 数据完整度；
- 主要风险；
- `适合 / 小仓尝试 / 继续观察 / 不适合 / 数据不足` 五档结论。

禁止使用“稳赚”“必涨”“无风险套利”等表达。第一版优先使用可解释规则和情景压力测试，不在缺少历史样本时训练复杂价格模型。

### 4.4 饰品 DIY 与贴纸搭配顾问

用户可以指定：

- 枪皮；
- 磨损和模板；
- 预算；
- 风格，例如黑金、极简、电竞、二次元、同色、撞色；
- 贴纸数量；
- 是否接受四连/五连、重复贴纸、刮擦贴纸；
- 更偏好视觉效果、性价比还是转售流动性。

推荐引擎应使用结构化贴纸目录，而不是让模型凭空生成贴纸名称。目录至少记录：

- 贴纸 ID、名称、系列和类别；
- 主色、辅助色、亮度和饱和度；
- 图案与风格标签；
- 普通、闪亮、全息、金色等版本；
- 图片来源；
- 当前价格与流动性；
- 可获得时记录刮擦效果。

预览能力分阶段实现：

1. 文本搭配方案和位置说明；
2. 第一版生成真实 CS2 检视代码；公开接口返回游戏渲染图时展示，否则明确返回代码模式；
3. 后续可选研究 CS2 Inspect Link 和 SteamDT 游戏级检视图，不作为 MVP 验收条件。

第一版所有候选组合先生成可验证的 CS2 masked inspect code。SteamDT 公开检视接口实际返回游戏渲染图时再展示并缓存图片；没有截图时只返回检视代码和限制说明，不使用通用画布坐标伪装真实贴纸槽位。

需要通过真实 API 试验确认：

- SteamDT 是否接受本地生成的自编码虚拟 Inspect Link；
- 是否完整支持五张贴纸、自定义位置和旋转；
- 异步检视图是否必须使用公网回调地址；
- 本地 Agent 在没有公网服务时如何获取异步结果。

生成式图片只能作为风格灵感图，不得冒充实际游戏效果或作为购买依据。

## 5. 建议技术架构

```text
Codex / Qoder / Trae / 其他 Agent
                 │
                 ▼
              MCP Server
                 │
        ┌────────┴────────┐
        │ CS2 Item Core  │
        └────────┬────────┘
                 │
    ┌────────────┼─────────────┐
    ▼            ▼             ▼
SteamDT      Steam公开库存     本地知识库
Adapter       Adapter          与分析规则
    │            │             │
    └────────────┼─────────────┘
                 ▼
          SQLite Event Store
                 │
       ┌─────────┼──────────┐
       ▼         ▼          ▼
   CLI报告    可选Worker    通知适配器
```

核心实现统一使用 TypeScript 与 Node.js。本机已确认存在 Node.js、npm、pnpm 和 Git；第一版项目脚本优先使用 npm，避免要求普通用户额外安装包管理器。MCP 优先采用本地 `stdio` 传输，不要求用户开放端口或部署公网服务。

Agent 分层如下：

```text
用户自然语言
    ↓
Codex / Qoder / Trae
    ↓
CS2 Item Agent Skill
    ↓
标准 stdio MCP
    ↓
TypeScript Core
    ├── SteamDT 与 Steam 数据适配器
    ├── SQLite 与分析计算
    ├── 挂刀风险评估
    └── DIY 推荐与预览
    ↓
结构化证据
    ↓
模型解释与建议
```

核心程序产生事实和指标；共享知识库保存市场经验；Skill 规定工具选择、证据表达和拒绝过度判断的流程。没有专属 Skill 时，支持 MCP 的 Agent 仍可使用基础功能。

建议模块边界：

```text
src/
├── adapters/
│   ├── steamdt/
│   ├── steam_inventory/
│   └── notifications/
├── catalog/
├── event_store/
├── inventory_monitor/
├── market_analysis/
├── arbitrage/
├── diy_advisor/
├── reporter/
├── cli/
├── mcp/
└── worker/

skills/
└── cs2-item-agent/
    ├── SKILL.md
    └── references/

knowledge/
├── evidence-policy.md
├── market-analysis.md
├── liquidity.md
├── seven-day-arbitrage.md
└── sticker-styles.md

data/
└── .gitkeep

examples/
└── anonymized/
```

不同 Agent 客户端只提供各自的 MCP 配置示例，底层 Core 和 MCP 不重复实现。预编译程序和 Docker 放到功能稳定后的发布阶段，第一版面向已经安装 Git 与 Node.js 的用户。

## 6. 数据存储初稿

SQLite 第一版建议至少包含：

- `data_sources`：数据源状态、最近成功时间和错误；
- `market_items`：饰品基础目录；
- `market_snapshots`：平台价格、在售和求购快照；
- `market_klines`：SteamDT 返回的 K 线缓存；
- `inventory_watches`：SteamID 监控任务；
- `inventory_snapshots`：库存快照元数据；
- `inventory_assets`：每次快照中的饰品；
- `inventory_events`：新增、移除和未知事件；
- `item_fingerprints`：可计算的饰品复合指纹；
- `arbitrage_evaluations`：挂刀评估输入、结果和模型版本；
- `diy_recipes`：贴纸组合、布局、预算和预览缓存；
- `alert_rules`：用户告警规则；
- `notifications`：通知发送与去重记录。

所有历史观察采用追加式写入。不得为了保存“最新状态”覆盖或删除历史证据。

## 7. 饰品身份与指纹原则

同类饰品统计使用标准化的 `market_hash_name` 或内部目录 ID，不需要为每个类目预先生成单件指纹。

单件饰品在字段可获得时使用复合指纹：

```text
defindex
+ paintindex
+ paintwear 原始位值
+ paintseed
+ quality
+ StatTrak/Souvenir
+ original_item_id（存在时）
```

规则：

- 显示出来的磨损小数不能作为唯一标识；
- `assetid` 不能默认是跨账号永久标识；
- 贴纸、名称标签、StatTrak 击杀数可能变化，只作为证据保存；
- 指纹可以连接已观察记录，不能凭空反查未知持有人；
- 指纹冲突时必须保留多个候选并降低置信度。

## 8. API、缓存与调度规范

- 所有外部请求必须设置超时、有限重试和指数退避；
- 严格遵守 SteamDT 与 Steam 的频率限制；
- 项目负责人已确认允许在用户本地保存 SteamDT 价格快照用于个人分析；仓库只发布适配器和脱敏样例，不再分发 SteamDT 的批量真实数据；
- 对批量行情、基础目录、K 线和检视图设置不同缓存周期；
- 同一轮任务尽量合并批量请求；
- 每条数据保存 `source`、`observed_at`、`source_updated_at` 和原始字段摘要；
- 适配器只负责字段映射，不把平台字段直接泄漏到核心领域模型；
- 接口变更不能静默吞错，必须记录为数据源异常；
- Steam 官方市场未文档化的公开接口只能作为可降级的辅助来源，必须低频缓存并允许关闭；
- 不使用需要用户 Cookie、密码或逆向登录态的高风险接口作为默认能力。

## 9. 安全、合规与开源规范

- 不获取 Steam 密码、短信验证码或平台 Cookie；
- 不绕过私密和好友可见库存；
- 不进行自动买入、卖出、报价或资金操作；
- 不提供操纵市场、骚扰账号或去匿名化能力；
- 公开仓库只包含适配器、规则、测试和脱敏样例；
- Valve、Steam 和第三方平台素材不得因代码开源而被默认视为可再分发；
- 枪皮和贴纸图片优先在本地按合法来源获取并缓存，不未经确认打包进仓库；
- 对外说明必须明确数据覆盖范围和不确定性；
- 第三方 API 免费政策变化时，项目仍保留替换数据源的能力。
- 开源许可证确定为 MIT。

## 10. 测试与质量要求

第一版优先保证 CLI、核心计算和测试，不开发大型前端。

必须测试：

- SteamDT 成功响应、错误响应、限流和字段缺失；
- 公开库存、私密库存、空库存和请求失败；
- 使用项目负责人的真实 SteamID 进行本地集成测试，但真实 ID 不得进入 Git、固定测试样本、日志、截图或演示数据；
- 快照差异与重复事件去重；
- 价格、手续费和收益计算精度；
- 乐观、基准、悲观情景边界；
- 过期数据不会生成高置信建议；
- 通知幂等与失败重试；
- DIY 候选不引用不存在的贴纸；
- 敏感信息不会进入日志、报告或测试快照；
- MCP 输出与 CLI 输出使用相同核心结果。

每个阶段完成后必须运行测试并报告：

- 实际执行命令；
- 通过和失败数量；
- 未覆盖风险；
- 仍需真实数据验证的能力。

## 13. 已确认的项目决策

1. 项目名称为 **CS2 Item Agent**，仓库建议名为 `cs2-item-agent`。
2. 核心使用 **TypeScript + Node.js**，不引入 Python 运行时依赖。
3. 核心保持跨平台，第一版在当前 Windows 环境重点验证。
4. 产品以用户按需查询和决策讨论为主，持续运行与通知属于可选能力。
5. 首个通知渠道为企业微信 Webhook。
6. 用户主动启用库存监控时，默认间隔为 30 分钟并允许修改。
7. 挂刀费率提供可修改模板，所有结论展示实际采用的费率。
8. 挂刀只输出证据、情景和风险等级，不承诺收益，不执行自动交易。
9. DIY 第一版要求推荐、价格和真实检视代码；游戏渲染图按公开接口实际能力提供。
10. 开源许可证为 MIT。
11. 仓库只提供虚构或脱敏 SteamID；项目负责人真实 ID 只用于本地集成测试。
12. 第一版完全不做匿名遥测。
13. 项目负责人已确认 SteamDT 数据可以在用户本地保存快照用于个人分析；仓库不发布批量真实数据。
14. SteamDT 为第一版默认行情源，但适配器必须可以替换。
15. MCP 第一版采用本地 `stdio`，不同 Agent 只提供配置适配层。
16. 预编译程序和 Docker 在项目功能稳定后补充。

## 26. 库存估值与高价值异动完善

在阶段 8 后补齐公开库存的本地估值与高价值异动边界。估值只针对完整成功快照，使用 BUFF 精确 `marketHashName` 基础类目最低在售价；同时配置数据源时优先 SteamDT，只配置 CSQAQ 时使用其个人批量价格接口，不在一次估值中静默混用来源。

### 26.1 数据与计算口径

- SteamDT 新增经过契约校验的 `/open/cs2/v1/price/batch` 适配器，单批最多 100 个名称并遵守每分钟一次限制；
- CSQAQ 保持个人批量价格单批最多 50 个名称；
- 本地价格缓存默认 30 分钟，跨进程保存 SteamDT 一分钟冷却标记；
- 缺失价格保持未知，输出按数量和唯一类目计算的两种覆盖率；
- 按本轮价格计算库存构成变化，按上轮数量计算市场价格影响；
- 特殊模板、极限磨损、贴纸和名称标签等实例溢价不计入基础估值。

SQLite migration 9 新增 `inventory_valuations`、`inventory_valuation_items` 与 `high_value_inventory_events`。估值、逐类价格、覆盖率、变化拆分、限制说明与高价值事件均追加保存在用户本机。

### 26.2 默认告警边界

- 单件基础价不低于 ¥1,000：只保存高价值物品事件；
- 库存构成估值变化绝对值不低于 ¥10,000，且相对上轮已知小计变化不低于 20%：满足总异动金额和比例条件；
- 前后价格覆盖率均不低于 90%：才允许标记总高价值库存异动；
- 纯市场价格变化不解释为库存转入、转出、买入或卖出。

默认值可通过本地 `.env` 修改，MCP `health_check` 返回实际采用参数。企业微信优先显示监控标签，否则使用掩码 SteamID。

### 26.3 CLI、MCP 与验证

- `inventory check` 和 `check_public_inventory` 在配置市场 API 后自动生成估值；
- 新增 `inventory valuation <steamId>` 和 `query_latest_inventory_valuation`，只读取本地最近估值；

## 27. 用户自定义组合告警

阶段 8 后新增确定性组合告警，不让模型直接决定是否触发。Agent 将自然语言转换成 JSON 草案，核心程序负责校验、保存、计算与通知。

### 27.1 规则与证据

- 表达式树支持 `all`/`any`，最多 4 层和 20 个叶子条件；

## 29. 行情交易与挂刀执行双模型

市场案例校准后，原有综合决策与挂刀计算被明确拆成两个互不替代的模型。两者可共享 API 观察，但分别形成结论，禁止把行情潜力当成挂刀可执行性。

### 29.1 `market_trading`

- 新增纯函数行情交易模型，组合单品趋势、板块相对强弱、提供方名义存量、专家估算有效流通盘、CSQAQ 监控样本集中度、大商运作标注和汰换关系；
- 板块、有效流通盘和大商经验必须携带 `sourceType`、来源标签和可选观察时间，并保存在 `expertAnnotations`，不冒充 API 事实；
- 单品与板块窗口不一致时选择最近窗口并披露误差；
- 只有完整结果概率与价格篮子才计算汰换静态期望，概率不等于 1 时不自动归一化；
- 输出支持信号、风险信号、未知和判断失效条件，不确认吸筹、洗盘、出货或控盘。

CLI 新增 `market trade`，`market decide` 保留为兼容别名；MCP 新增 `analyze_market_trading`，`analyze_item_decision` 保留为兼容别名。行情报告追加保存为 `decision_reports.report_type=market_trading`。

### 29.2 `hanging_execution`

- 所有挂刀报告显式返回模型身份、用途和非适用范围；
- 自动候选池优先武器箱、允许普通枪皮，默认排除贴纸/布章、刀、手套和未知品类；
- 用户明确指定被排除品类时仍计算费率和七日情景，但保留品类警告，且不自动升级为候选；
- 筛选报告披露提供方候选数、准入数与默认排除数。

### 29.3 Skill 与验证

- 专属 Skill 新增 `model-routing.md`，要求行情与挂刀分别调用、分别回答，禁止平均为一个分数；

## 31. 发布基线

2026-07-22 开始建立首个可回滚的源码发布基线，不改变阶段 0 至阶段 8 已确认的架构和产品边界。

- 预发布版本统一为 `0.8.0-alpha.1`；
- 当前能力基线为 42 个 MCP 工具、12 个 SQLite 迁移和 77 项自动化测试；
- CI 在 Windows 与 Linux 的 Node.js 24 上运行 `npm ci` 和 `npm run release:verify`；
- `release:verify` 统一执行严格类型检查、构建、完整测试与 MCP 协议冒烟；
- `.agents/skills/cs2-item-agent` 是专属 Skill 的唯一权威来源，客户端适配入口由脚本生成并在发布验证中检查一致性；
- Codex 使用 `.codex/config.toml`，Claude Code 与 Qoder 共用根目录 `.mcp.json`，WorkBuddy 使用 `.workbuddy/mcp.json`；所有项目级 MCP 配置统一调用 `scripts/run-mcp.mjs`，不保存绝对路径或密钥，并由 `mcp:config:check` 纳入发布验证；
- Trae Windows 实机界面已确认从项目根目录 `.trae/mcp.json` 加载项目级 MCP；仓库提交该配置，用户首次打开项目时只需开启“启用项目级 MCP”，不再手动填写服务器命令和绝对路径；
- 客户端专门适配范围收敛为 Codex、Claude Code、Qoder、WorkBuddy 和 Trae；其他支持本地 stdio MCP 的客户端只保留通用配置兼容，不继续增加客户端专属核心或目录；
- `scripts/run-mcp.mjs` 作为唯一自动启动器：Node.js 24 仍是外部前置条件；启动器自动创建且不覆盖 `.env`、按 `package-lock.json` 执行可复现依赖安装、按源码新旧决定构建、串行执行幂等数据库迁移，并把所有准备日志限制在 stderr；
- MCP 初始化说明与 `health_check` 自带中文 `usageGuide` 和动态 `configurationGuide`；Agent 可按当前进程读取状态解释各变量启用的能力、填写位置、重启要求和安全验证方式，但不读取、回显或索取配置值；
- `.local/mcp-bootstrap.lock` 防止多个 Agent 客户端首次打开同一仓库时并发修改依赖或数据库；首次准备超过客户端等待时间时可用 `npm run setup` 预热，不改变最终 MCP 启动入口；
- `.env`、本地数据库、缓存、图片、真实 SteamID 和构建产物继续排除在 Git 基线之外；
- 首个基线仍是源码预发布版本；预编译程序、Docker 和正式发行包继续属于后续安装体验里程碑。
